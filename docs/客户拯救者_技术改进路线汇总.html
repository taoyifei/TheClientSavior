技术改进路线｜The Client Savior

客户拯救者：后续优化与技术演进汇总

本页面将现有业务改进点、上线困难点和技术优化方向整合为一份可展示的 HTML 方案， 重点突出语音转文字、政策知识库扩展、混合检索、业务规则保护、本地大模型部署、网格通数据接口和安全上线路径。

当前形态：FastAPI + React 工作台 核心链路：政策召回 + LLM + 兜底模板 演进重点：找准政策、防止错推、可内网上线

一、当前状态与已有能力

当前系统已经具备演示级智能体闭环，但仍处于“轻量 MVP”阶段，后续重点是从演示版走向可上线的业务系统。

当前架构

前后端分离工作台

主界面已迁移为 FastAPI 后端 + React/Vite/TypeScript + Ant Design 前端，旧 Streamlit 仅作为 legacy 备份。

核心链路

召回 → LLM → 兜底

投诉文本与客户画像进入本地政策召回，再交给云端大模型生成分析、推荐政策和四段式挽留话术；模型失败时使用本地模板兜底。

演示边界

本地 JSON + 内存看板

当前政策库、演示案例和客户画像以本地 JSON 为主，风险看板以进程内存记录为主，适合培训演示，不适合长期生产使用。

二、已识别的业务改进点

这些是当前业务侧最直接的改进需求，后续技术路线应围绕它们展开。

输入效率

新增语音转文字

客服、网格和相关部门之间经常直接转发语音文件，人工转文字耗时耗资源；应支持录音或上传音频后自动转写为投诉文本。

政策覆盖

导入更多套餐与优惠包

当前只有少量政策套餐，实际可购买基本套餐有几十种；此外优惠包维度尚未充分覆盖，需要扩展到套餐之外的权益、包、合约和返费政策。

数据自动化

对接网格通客户画像

手机号、月租、网龄、家庭号码数等信息目前仍需手动输入，后续应通过内部接口自动回填，减少客服录入成本。

场景扩展

扩展有线家宽、BC/B/H 端

当前主要处理 C 端无线投诉，后续需扩展到有线宽带、BC 融合、B 端和 H 端业务场景，并引入相关部门配合。

三、推荐的技术演进总架构

建议后续不要只追求“模型更大”，而是建设“政策知识库 + 混合检索 + 规则保护 + 模型网关 + 数据闭环”。

客户输入手机号 / 投诉文本 / 语音
画像服务网格通 / 客户标签 / 超耗状态
政策知识库套餐 / 优惠包 / 宽带 / 终端 / 商客
混合检索关键词 + 向量 + 结构化过滤
LLM 生成云端 / 本地模型 / 模板兜底
核心判断：政策推荐类产品最关键的是“找准政策”和“防止错推”。因此，向量数据库与业务规则保护的优先级应高于本地大模型部署。

四、技术优化方向详解

P1

1. 混合检索 + 向量数据库

当政策数量从十几条扩展到几十或上百条时，单纯关键词匹配会召回不全、排序不稳。建议升级为：

  • 关键词召回：保障精确词命中。
  • 向量召回：理解“月底超量”“刷视频不够”等语义表达。
  • 结构化过滤：按月租、网龄、是否宽带、是否降档意图过滤。
  • 重排序：结合业务规则和客户价值重新排序。
P0

2. 结构化政策知识库

将政策从普通 JSON 卡片升级为统一 OfferItem 数据结构，记录适用客群、办理条件、互斥规则、合约风险、有效期和来源文档。

  • 支持主套餐、优惠包、宽带、FTTR、亲情网、终端、商客融合。
  • 支持从 Word / Excel 中抽取政策卡。
  • 支持后续政策有效期和版本管理。
P0

3. 业务规则保护层

大模型不能直接决定最终推荐，必须有规则层约束候选政策和模型输出。

  • 未明确降档时，不推荐更便宜主套餐。
  • 涉及预存、合约、宽带安装、终端库存，必须提示以系统办理结果为准。
  • LLM 输出后再次校验，避免模型绕过规则。
P2

4. 本地化大模型部署

本地大模型适合中长期内网上线，不建议作为下一步唯一重点。建议通过统一模型网关支持：

  • 演示阶段：云端 Qwen API，保证效果和速度。
  • 内网阶段：vLLM / Ollama / 私有推理服务。
  • 统一 OpenAI-compatible 接口，减少业务代码改造。
P1

5. 网格通客户画像接口

后续通过手机号自动获取客户画像，提升推荐准确率。

  • 当前套餐、月租、网龄、上月流量、是否超耗。
  • 宽带状态、家庭号码数、历史投诉次数。
  • 合约状态、终端分期、客户价值标签。
P1

6. 语音转文字入口增强

第一版建议只做“录音/上传音频 → ASR 转文字 → 客服确认 → 生成方案”，不要一开始做实时 WebSocket 通话系统。

  • 提升客服输入效率。
  • 符合现有语音转发工作流。
  • 转写结果必须可编辑，避免识别错误直接进入推荐。
P2

7. 风险看板持久化与权限控制

演示版内存看板适合现场展示，但上线后需要数据库、权限和审计。

  • SQLite / MySQL / PostgreSQL 存储客户处理记录。
  • Redis 支持热点缓存和任务状态。
  • 风险看板仅内网访问，操作记录可审计。
P3

8. 多业务域扩展

将系统从 C 端无线投诉扩展到有线家宽、BC 融合、B 端和 H 端场景。

  • 不同业务域独立政策库和规则集。
  • 投诉内容自动路由到业务域。
  • 跨部门协作维护政策和办理口径。

五、优先级矩阵

建议按“推荐正确性 → 数据接入 → 检索能力 → 安全上线 → 模型私有化”的顺序推进。

优先级 方向 原因 建议产出
P0 业务规则保护层 直接防止模型推荐错误政策，是上线前必须做的安全阀。 降档规则、合约风险、办理条件强提醒、输出后校验。
P0 结构化政策知识库 后续导入更多套餐、优惠包、宽带和终端政策的基础。 OfferItem 数据结构、政策来源、互斥规则、有效期管理。
P1 混合检索 / 向量数据库 政策数量增加后，提升召回率和匹配准确率。 Embedding、向量库、关键词+向量召回、重排序。
P1 网格通客户画像接口 减少人工录入，提高推荐对客户实际状态的适配度。 手机号查画像、超耗状态、客户价值、宽带/合约状态。
P1 语音转文字 演示效果明显，且符合真实语音转发的工作流。 ASR 转写、可编辑文本、投诉框自动填充。
P2 数据库持久化与权限控制 从演示走向上线必须解决记录丢失、审计和安全访问问题。 客户案例表、推荐结果表、风险队列表、操作日志。
P2 本地大模型部署 解决数据出网、内网可用和成本可控，但不应优先于政策与规则。 模型网关、vLLM/Ollama、OpenAI-compatible 服务。
P3 多业务域扩展 需要跨部门协作和更多政策数据，适合中长期推进。 有线家宽、BC/B/H 端政策库和规则集。

六、分阶段实施路线

第 1 阶段
1-2 周

先把推荐做稳

完善业务规则保护层、政策卡结构、降档规则、合约和风险提醒；补齐更多套餐与优惠包样例，建立测试案例集。

第 2 阶段
1 个月

引入语义检索和语音入口

上线向量化政策库 MVP，支持语义匹配；新增语音转文字入口，把语音投诉转成可编辑文本。

第 3 阶段
2-3 个月

连接真实画像与看板闭环

对接网格通或内部客户画像服务；风险看板改成数据库持久化，并增加权限控制和操作审计。

第 4 阶段
3-6 个月

内网部署与多业务扩展

通过模型网关支持本地大模型，扩展有线家宽、商客融合、B/H 端场景，形成多业务域智能投诉处理平台。

七、可落地的向量数据库方案

建议从小规模试点开始，不要一上来做复杂 RAG 平台。

模块 建议方案 说明
政策文本切分 按政策卡粒度，不按长文档切碎 每条政策一张卡，包含标题、适用客群、权益、条件、风险提醒，便于可解释推荐。
Embedding bge-m3 / 通用中文 embedding 将政策卡和客户投诉转成向量，支持“语义相近但关键词不同”的召回。
向量库 Milvus / Qdrant / FAISS 演示或轻量版可用 FAISS；上线和多用户场景建议 Milvus 或 Qdrant。
检索策略 关键词 + 向量 + 结构化过滤 先召回候选，再根据月租、是否降档、是否宽带、客户类型过滤不合适政策。
重排序 业务分 + 语义分 + 风险规则 不要只按向量相似度排序，必须加入政策适配度和业务安全规则。
投诉文本 → 关键词召回 TopN → 向量召回 TopN → 合并去重 → 结构化规则过滤 → 业务重排序 → LLM 生成推荐理由和客服话术 → 输出后规则校验

八、本地大模型部署建议

本地模型不是替代政策库和规则层，而是解决“内网可用、数据不出网、成本可控”的能力。

阶段一

继续使用云端模型

短期为了演示效果和速度，继续使用云端 Qwen API，配合本地模板兜底和超时控制。

阶段二

抽象模型网关

统一 OpenAI-compatible 接口，支持在云端模型、本地 Ollama、vLLM 服务之间切换。

阶段三

内网 GPU 服务器部署

正式上线时可在内网部署 Qwen 系列模型，通过权限、审计和脱敏策略控制敏感数据。

建议：优先建设“政策库 + 检索 + 规则”，再做本地大模型。否则模型即使本地化，也可能因为政策召回不准而推荐错误。

九、上线难点与风险控制

组织协同

跨部门配合难度高

上线需要在线公司、客服部、网络部、网管中心、市场部、数智化部等协作，涉及投诉数据、网络权限、套餐政策和网格通接口开通。

数据安全

风险看板应限制内网访问

风险看板包含客户投诉和手机号等敏感信息，后续上线应做到内网访问、脱敏展示、权限分级和操作审计。

十、可直接用于汇报的总结话术

当前系统已经完成从 Demo 到前后端工作台的第一步，具备号码查询、客户画像、政策推荐、LLM 生成、风险看板和本地兜底能力。 后续技术演进的重点不是单纯把模型换大，而是建立结构化政策知识库、混合检索和业务规则保护层，确保推荐政策“找得准、讲得清、不出错”。 在此基础上，再逐步接入网格通客户画像、语音转文字、数据库持久化和本地大模型部署，最终形成一个可内网上线、可审计、可扩展到多业务域的客户挽留智能工作台。
