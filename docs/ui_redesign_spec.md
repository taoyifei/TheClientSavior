# 客户拯救者 UI/UX 重构设计规格

## 1. 产品定位

客户拯救者是面向一线客服和现场经理的客户挽留工作台，目标是在客户投诉进入后，快速完成号码查询、客户画像识别、超耗/离网风险判断、推荐业务生成、话术输出和后台跟进。

## 2. 用户角色

- 一线客服：快速输入号码、投诉，拿到可执行话术。
- 现场经理：查看高风险客户、跟进优先级、推荐业务分布。
- 培训评委：能一眼看出 LLM 是核心能力，产品可落地。

## 3. 核心体验原则

- 号码优先：手机号查询永远是页面第一入口。
- 决策优先：是否超耗、风险等级、推荐业务、跟进优先级必须最醒目。
- 行动优先：首推业务和话术要比技术状态更醒目。
- 管理闭环：后台看板要体现客户跟进队列。
- 演示友好：三步完成一次演示，不能让用户找按钮。

## 4. 页面信息架构

- 顶部 App Shell：产品名、运行状态、关键统计。
- 工作台主导航：客服工作台、风险看板、政策库。
- 号码查询 Command Bar：手机号查询、清空、演示案例快捷入口。
- 客户 360 Snapshot：本地客户画像、用量、超耗、宽带和携转信号。
- AI 决策驾驶舱：首推业务、超耗、风险、投诉类型、情绪、优先级。
- 投诉录入与画像补充：投诉原文和画像参数。
- 推荐方案与话术：生成状态、首推业务、Top 3 政策、四段式话术。
- 后台风险队列：KPI、高风险客户、筛选、导出和清空。
- 政策库：政策货架、搜索、分类筛选、政策卡。

## 5. Design System

### 色板

- 企业蓝：`#1677ff`
- 深海军蓝：`#0b2f6b`
- 正文深色：`#101828`
- 辅助文字：`#667085`
- 背景浅灰：`#f5f8fc`
- 风险红：`#d92d20`
- 预警橙：`#f79009`
- 成功绿：`#12b76a`
- 边框线：`rgba(16, 24, 40, 0.10)`

### 字体

- 使用系统字体栈，不依赖外部字体。
- 标题强调业务对象，KPI 强调数字和状态。
- 正文保持 14px-16px，避免视觉噪声。

### 间距

- 页面大区块间距：18px-24px。
- 卡片内边距：16px-22px。
- 表单控件间距：10px-14px。
- 移动端自动单列，减少横向拥挤。

### 圆角

- 大容器：24px。
- 中型卡片：16px。
- Badge / Chip：999px。

### 阴影

- 小阴影：`0 4px 14px rgba(16, 24, 40, 0.06)`。
- 中阴影：`0 12px 32px rgba(16, 24, 40, 0.10)`。
- 不使用厚重投影，避免像营销页。

### Badge

- 用于状态、风险等级和模式。
- 颜色语义固定：蓝色代表系统/模型，红色代表风险，橙色代表预警，绿色代表成功。

### Card

- 每张卡只表达一个业务意图。
- 重要卡片直接展示核心信息，不把主信息藏在 expander。

### KPI

- 使用标签、主值、说明三段式。
- 主值字体更重，辅助说明弱化。

### Button

- 查询客户、生成方案是主按钮。
- 清空、下载、筛选是次级按钮。
- 按钮必须有明显 hover 和 focus 状态。

### Table

- 风险队列保留 dataframe，但外层用产品化标题、说明和筛选条。
- 默认高风险在前，不展示内部排序字段和完整手机号。

### Empty State

- 未生成前展示流程引导。
- 未查询到客户时提示可继续手动补充画像。

### Warning State

- LLM 失败、手机号格式错误、超耗/携转风险使用明确预警样式。

## 6. 反模式

- 不要大段 Markdown 结果。
- 不要把推荐政策藏在 expander 里。
- 不要让号码查询低于演示案例。
- 不要用过多渐变。
- 不要让按钮视觉权重一样。
- 不要让后台看板只是 dataframe。
- 不要 AI 紫粉渐变、霓虹、堆叠 emoji。
- 不要无层级地堆表单字段。

## 7. 验收清单

- 打开页面第一眼能看到号码查询。
- 未生成前也能看到决策摘要占位。
- 生成后 3 秒内能看懂首推业务。
- 高风险、超耗、携转意向有明显颜色。
- 手机号全程脱敏。
- 页面在 1440、1024、768 宽度下可用。
- smoke_test.py 通过。
