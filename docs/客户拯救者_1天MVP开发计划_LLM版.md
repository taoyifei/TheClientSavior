# The Client Savior「客户拯救者」1 天可演示 MVP 开发计划（LLM 核心版）

> 目标：在 **1 天内**做出一个能在普通笔记本电脑上稳定演示的智能体 Demo。运行环境为 Ultra 9 CPU、无独立显卡。  
> 关键要求：**LLM 必须是核心卖点**，允许本地兜底，但正常演示链路必须调用 LLM 完成客户诉求理解、政策二次判断和个性化话术生成。  
> 性能要求：点击后不要出现 1–2 分钟等待；正常目标为 **8 秒内返回 LLM 结果**，超时自动兜底。

---

## 1. 产品定位

### 1.1 智能体名称

**The Client Savior｜客户拯救者**

### 1.2 一句话口号

**投诉秒变留人机会，政策套餐自动配对！**

### 1.3 目标用户

广东移动分公司网络经理、一线客服、网格经理、投诉协同处理人员。

### 1.4 要解决的问题

一线人员处理客户投诉时，经常遇到：

- 客户表达口语化，真实诉求不清楚；
- 套餐、宽带、终端、保有政策分散，人工检索慢；
- 新人不知道该推荐哪个政策，也不知道怎么安抚；
- 客户有携转/离网风险时，缺少即时挽留话术；
- 管理者难以快速看到投诉类型、推荐政策和风险分布。

---

## 2. MVP 结论：做「云端 LLM + 本地政策卡 + 规则兜底」

### 2.1 为什么不本地跑大模型

Ultra 9 CPU 无独显，现场还要保证响应速度。一天开发周期下，不建议本地部署 Qwen/Ollama/Transformers 模型，否则容易出现启动慢、推理慢、模型下载失败、现场网络/内存问题。

### 2.2 LLM 选择

MVP 默认接入 **云端 Qwen / DashScope / 阿里云百炼 OpenAI 兼容接口**。

推荐环境变量：

```bash
DASHSCOPE_API_KEY=你的API_KEY
LLM_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
LLM_MODEL=qwen-turbo
LLM_TIMEOUT=8
```

也可以换成其他 OpenAI-compatible 服务，只要支持 `chat.completions`。

### 2.3 LLM 在系统中的核心作用

本项目不是“规则匹配套壳”，而是把 LLM 放进主流程：

```text
投诉文本 + 客户画像
        ↓
本地快速召回候选政策 Top 5
        ↓
LLM 读取客户投诉、画像、候选政策
        ↓
LLM 输出：
1. 客户真实诉求结构化
2. 情绪与离网风险判断
3. Top 3 政策二次排序
4. 每个政策的推荐理由
5. 个性化安抚 + 挽留 + 办理话术
        ↓
页面展示 LLM 结果、政策卡、处理建议、看板
```

### 2.4 本地规则的定位

本地规则不是核心卖点，只负责三件事：

1. **快速召回**：从 10 张政策卡中先筛出 Top 5，减少 LLM 输入长度；
2. **稳定兜底**：API Key 缺失、网络失败、超时后仍能演示；
3. **可解释性**：给出关键词命中、客户画像命中等基础理由。

## 3. 一天内必须收敛的主流程

只做一条主流程，不做复杂平台。

```text
输入投诉文本 + 客户画像
        ↓
本地政策召回 Top 5，1 秒内完成
        ↓
调用 LLM 生成智能挽留方案，目标 3–8 秒
        ↓
如果 LLM 超时/失败，自动模板兜底
        ↓
展示推荐政策、个性话术、办理提醒、看板统计
```

## 4. 页面设计

### 4.1 Tab 1：投诉处理

#### 输入区

- 投诉内容文本框；
- 当前套餐月租；
- 客户类型：新入网 / 存量 / 高套 / 家庭用户 / 商客；
- 网龄年数；
- 是否有宽带；
- 是否有换机需求；
- 家庭移动号码数；
- 是否明确想离网；
- LLM 状态显示：在线 / 离线 / 未配置 Key。

#### 按钮

```text
生成 AI 挽留方案
```

按钮名称必须突出 AI/LLM，不要只写“匹配”。

#### 输出区

展示 5 块内容：

1. **LLM 客户画像识别**
   - 投诉类型；
   - 关键诉求；
   - 情绪状态；
   - 离网风险；
   - 推荐处理策略。

2. **AI 推荐政策 Top 3**
   - 政策名称；
   - 匹配分；
   - LLM 推荐理由；
   - 核心卖点；
   - 办理提醒。

3. **个性化挽留话术**
   - 安抚开场；
   - 方案介绍；
   - 风险说明；
   - 下一步引导。

4. **办理提醒**
   - 资格需系统核验；
   - 是否有合约期；
   - 是否有首充/预存/调测费；
   - 是否有优惠到期恢复资费。

5. **本次 AI 调用信息**
   - 模型名称；
   - 耗时；
   - 是否命中缓存；
   - 是否使用兜底。

---

### 4.2 Tab 2：管理看板

使用 `st.session_state` 统计本次演示过程中的数据：

- 已处理投诉数；
- 高风险客户数；
- 平均 AI 响应耗时；
- 投诉类型分布；
- 推荐政策次数 Top 5；
- LLM 成功次数 / 兜底次数。

展示方式：

- `st.metric`；
- `st.bar_chart`；
- 简单表格。

不需要数据库。

---

### 4.3 Tab 3：政策库

展示本地 `policies.json`：

- 搜索框；
- 分类筛选；
- 展开政策卡查看详情；
- 显示适用客户、权益、办理条件、风险提醒、话术提示。

## 8. LLM Prompt 设计

### 8.1 LLM 输入

每次只传必要内容，避免慢：

```text
客户投诉：不超过 800 字
客户画像：套餐、网龄、是否宽带、是否换机、家庭号码数、是否想离网
候选政策：本地召回 Top 5，每张只传标题、适用场景、权益、条件、风险提示、规则分
```

### 8.2 LLM 输出必须是 JSON

建议输出结构：

```json
{
  "customer_analysis": {
    "complaint_type": "流量不足/资费争议/宽带质量/换机需求/离网风险/其他",
    "emotion": "平稳/不满/强烈不满",
    "risk_level": "低/中/高",
    "key_needs": ["更多流量", "降低月租"],
    "summary": "客户主要因为流量不够且觉得套餐贵而投诉。"
  },
  "recommended_policies": [
    {
      "policy_id": "P02",
      "rank": 1,
      "reason": "客户流量使用诉求强，愿意小幅升档时该方案权益提升明显。",
      "talking_point": "重点强调流量提升、当天生效和亲情网叠加。"
    }
  ],
  "retention_script": {
    "opening": "先安抚客户情绪的话术。",
    "solution": "结合推荐政策进行解释的话术。",
    "risk_disclosure": "合约、首充、优惠到期等提醒。",
    "next_step": "引导客户授权查询或办理。"
  },
  "internal_notes": ["先查是否目标客户", "以系统办理结果为准"]
}
```

## 9. LLM 性能控制

为了不出现 1–2 分钟等待，必须做这些限制：

| 控制项 | 要求 |
|---|---|
| 单次 LLM 调用 | 主流程只调用 1 次 |
| 超时时间 | 默认 8 秒，最多 10 秒 |
| 最大输出 | `max_tokens` 600–900 |
| 候选政策 | 最多传 Top 5 |
| 温度 | `temperature=0.2~0.4` |
| 缓存 | 对同一投诉 + 画像 + 候选政策做 hash 缓存 |
| 失败处理 | 立即走本地模板兜底 |
| UI 展示 | 明确显示「LLM 生成 / 模板兜底」状态 |

推荐策略：

```text
点击按钮后：
1 秒内显示本地召回结果
随后调用 LLM
8 秒内成功则展示 LLM 方案
8 秒失败则展示兜底方案，并标记“LLM 超时，已启用本地模板”
```

---

## 10. 兜底策略

LLM 失败时不能假装是 LLM 结果，要明确显示：

```text
本次 LLM 未成功返回，已启用本地模板兜底。
```

兜底输出：

- 用本地规则识别投诉类型；
- 用本地 Top 3 政策卡；
- 用模板生成话术。

这样现场不会翻车，但 LLM 在线时仍然是核心演示链路。

## 13. 现场汇报话术

### 13.1 开场

> 我们做的是一个面向一线投诉处理的 AI 智能体，名字叫 The Client Savior 客户拯救者。它能把客户投诉自动转成客户画像和关键诉求，再结合本地政策库，由大模型生成个性化挽留方案。

### 13.2 技术亮点

> 系统不是简单关键词匹配。关键词召回只负责把候选政策缩小到 Top 5，真正的客户意图理解、风险判断、政策推荐理由和个性话术，是由 LLM 完成的。为了保证现场速度，我们只做一次短上下文 LLM 调用，并设置 8 秒超时和本地模板兜底。

### 13.3 业务价值

> 对一线人员来说，它可以减少查政策、想话术的时间；对新人来说，可以降低经验门槛；对管理者来说，可以看到投诉类型和政策推荐分布。

---

## 14. 成功标准

一天结束时，Demo 只要达到以下标准就算成功：

- 可以输入投诉并生成 AI 挽留方案；
- LLM 在线时能显示模型名、耗时、AI 生成结果；
- LLM 失败时能兜底，不影响演示；
- Top 3 政策推荐基本合理；
- 看板能统计处理数量、高风险数量、推荐政策次数；
- 全流程点击后不会超过 10 秒；
- UI 看起来像一个完整产品，而不是代码实验。
