# 客户拯救者政策匹配机制说明

本文说明“客户拯救者 The Client Savior”中政策匹配是怎么实现的。当前版本不使用向量库、不使用 LangChain、不使用本地大模型，匹配链路采用“本地规则召回 + 云端 LLM 生成”的轻量架构。

## 1. 匹配链路总览

一次生成请求的主流程如下：

```text
前端提交投诉内容 + 客户画像
→ FastAPI /api/generate
→ ClientSaviorAgent.run()
→ match_policies() 本地规则召回 Top 5
→ 将 Top 5 候选政策交给 LLM
→ LLM 只能从候选 policy_id 中选择 Top 3
→ 合并本地政策详情、分数、原因
→ 返回客户分析、推荐政策、四段式话术、内部提醒
```

关键点：

- 手机号不进入匹配和 LLM prompt，只用于本地客户查询与后台看板。
- 本地规则负责“候选召回”，保证 LLM 只在可控政策范围内选择。
- LLM 负责“话术生成、推荐理由表达和客户分析”，但不能编造政策。
- LLM 失败或无 API Key 时，本地模板兜底仍使用同一批本地 Top 5 候选。

## 2. 数据输入

匹配函数入口位于 `src/matcher.py`：

```python
match_policies(
    complaint: str,
    profile: CustomerProfile,
    policies: list[dict[str, object]],
    top_k: int = 5,
) -> list[PolicyMatch]
```

输入来源：

- `complaint`：客户投诉原文，例如“套餐太贵，不行我就携号转网了”。
- `profile`：客户画像，不包含手机号，字段包括月租、客户类型、网龄、是否有宽带、是否有换机需求、家庭号码数、是否携转等。
- `policies`：`data/policies.json` 中的静态政策卡。
- `top_k`：默认返回 5 个候选政策。

政策卡核心字段：

- `id`：政策编号，例如 `P05`。
- `title`：政策名称。
- `category`：政策分类。
- `price`：资费或办理价格。
- `target`：适用客户。
- `keywords`：召回关键词。
- `benefits`：核心权益。
- `conditions`：办理条件。
- `risk_notes`：风险提醒。
- `script_hint`：客服话术提示。

## 3. 本地规则打分

每张政策卡都会经过 `_score_policy()` 计算匹配分。初始分为 0，命中不同条件会累积分数，同时记录匹配原因。

### 3.1 投诉关键词命中

系统会把客户投诉原文与政策卡的 `keywords` 做包含匹配。

规则：

```text
命中关键词数 × 8 分，最高 40 分
```

示例：

客户说“流量不够，刷视频总是超量”，如果某政策关键词包含“流量”“不够”“刷视频”，则该政策会获得关键词命中分，并记录“命中关键词”原因。

### 3.2 携转/离网风险

系统会从两个方向判断是否有离网风险：

- 客户画像中 `wants_port_out=True`。
- 投诉原文包含携转、离网、转网、不用、换别家等表达。

如果客户存在离网风险，并且政策属于保有类政策，则加分：

```text
保有类政策 +25 分
高风险保有抓手额外 +15 分
```

这会让无忧包、返费、保有类政策在高风险客户场景中更靠前。

### 3.3 当前月租档位

系统会根据当前套餐月租做粗分层：

- `monthly_fee <= 39`：更偏向低门槛、大流量、升档类方案，相关政策加 20 分。
- `monthly_fee >= 119`：更偏向中高套保有、宽带融合类方案，相关政策加 20 分。

这样可以避免低月租客户优先推荐高门槛保有包，也能让高套客户更容易匹配到保有和家庭融合政策。

### 3.4 宽带与网络质量诉求

如果客户画像中已有宽带，或者投诉中出现宽带、WiFi、网速、卡顿、上网课等表达，则宽带类政策加分：

```text
宽带/FTTR/全家享相关政策 +25 分
```

典型命中场景：

- “家里 WiFi 到卧室没信号”
- “宽带晚上特别卡”
- “孩子上网课经常断”

### 3.5 换机与终端诉求

如果客户画像中 `wants_device=True`，或者投诉中出现手机、换机、购机、终端、分期等表达，则终端购机类政策加分：

```text
终端/购机/预存相关政策 +30 分
```

典型命中政策是终端购机金币、预存抵扣等方案。

### 3.6 商客诉求

如果客户类型是“商客”，或者投诉中出现商户、门店、宣传、工作号、AI 监控等表达，则商客融合政策加分：

```text
商客融合/AI企计划类政策 +30 分
```

这类规则用于识别门店经营、宣传触达、企业宽带等场景。

### 3.7 家庭融合诉求

如果家庭移动号码数 `family_mobile_count >= 3`，家庭融合和亲情网类政策加分：

```text
家庭融合/亲情网政策 +20 分
```

用于识别多号码家庭、省钱、互打、亲情网等场景。

### 3.8 网龄保有倾向

如果客户网龄 `tenure_years >= 5`，且政策属于保有类政策，则加分：

```text
老客户保有政策 +15 分
```

这体现“老客户优先保有”的业务逻辑。

### 3.9 资格门槛修正

部分政策有典型门槛，系统会做轻量扣分，不直接过滤，避免过早丢掉候选：

```text
P04 1000元充送无忧包：当前月租 < 199 时 -35 分
P05 600元充送无忧包：当前月租 < 119 时 -30 分
```

原因：

- 演示版不连接真实 BOSS/CRM 系统，不能做最终资格判断。
- 扣分比硬过滤更稳妥，仍允许 LLM 或客服把它作为备选核查。
- 页面和话术中会提示“办理资格以系统办理结果为准”。

## 4. 政策类型识别

政策类型不是靠数据库标签表，而是通过轻量文本规则判断。

系统会把政策的这些字段合并成搜索文本：

```text
id + title + category + target + script_hint
```

然后用关键词判断政策类型：

- 保有类：保有、返费、无忧包、老客户。
- 流量/升档类：流量、低资费、升档、畅玩。
- 宽带类：宽带、FTTR、WiFi、全家享。
- 终端类：终端、换机、购机、手机。
- 家庭类：家庭、亲情、全家享、副号。
- 商客类：商客、商户、门店、慧商、AI企计划。

这套方式的优点是简单、透明、可解释，适合一天内可演示的 MVP。

## 5. 排序与候选召回

每张政策计算完成后，会得到：

- `score`：本地匹配分。
- `reasons`：本地匹配原因。
- `policy`：完整政策卡。

排序规则：

```text
先按 score 降序
同分时按 policy_id 升序
最后取 Top 5
```

返回结构是 `PolicyMatch`，后续会进入智能体主流程。

## 6. LLM 如何使用匹配结果

本地召回不是最终答案，而是给 LLM 的候选范围。

`ClientSaviorAgent.run()` 中的处理逻辑：

1. 调用 `match_policies(..., top_k=5)` 得到本地 Top 5。
2. 取出 Top 5 的完整政策卡作为 `candidate_policies`。
3. `src/llm_client.py` 会用 `compact_policy()` 压缩政策，只保留必要字段。
4. Prompt 明确要求 LLM：
   - 只能从候选政策 `id` 中选择。
   - `recommended_policies` 最多返回 3 个。
   - 输出必须是合法 JSON。
   - 不能编造政策。
   - 办理资格、合约、优惠期、预存、初装费、调测费必须提示“以系统办理结果为准”。

因此，LLM 的角色不是“从全量政策库自由搜索”，而是在本地规则召回的候选政策中做表达、排序和话术生成。

## 7. 结果合并与防编造

LLM 返回后，`src/agent.py` 会执行 `_merge_result()`。

合并时会做三件事：

1. 建立本地候选索引 `local_by_id`。
2. 只接受 `policy_id` 存在于本地 Top 5 的推荐项。
3. 把本地政策详情、分数和原因补回推荐结果。

如果 LLM 返回了不存在的政策 ID，会被丢弃。

如果 LLM 返回不足 3 条，系统会用本地匹配 Top 5 中尚未使用的政策补齐，保证页面至少有可展示的推荐政策。

最终前端拿到的推荐政策包含：

- `policy_id`
- `rank`
- `title`
- `reason`
- `talking_point`
- `local_score`
- `local_reasons`
- `policy`
- 拍平后的 `category`、`price`、`benefits`、`conditions`、`risk_notes`

## 8. LLM 失败时的兜底匹配

如果出现以下情况：

- 未配置有效 API Key。
- LLM 超时。
- LLM 返回非法 JSON。
- 网络异常。
- 兼容接口不支持部分参数。

系统不会卡死，而是走本地模板兜底：

```text
本地 Top 5 匹配结果
→ build_fallback_result()
→ 生成本地客户分析、推荐政策、四段式话术、内部提醒
```

兜底和 LLM 使用同一批本地匹配候选，因此结果仍然与客户投诉和画像相关。

## 9. 为什么不用向量库

当前 MVP 明确不使用向量库，原因是：

- 政策库只有十几张卡，规则召回已经足够快。
- 演示现场更需要稳定、可解释、可控。
- 向量库会增加依赖、初始化成本和部署复杂度。
- 政策匹配需要结合业务门槛，例如月租、网龄、宽带、携转风险，这些结构化条件用规则更直接。

后续如果政策库扩展到数百或数千条，可以升级为：

```text
结构化规则过滤
→ BM25/关键词召回
→ 向量召回
→ 规则重排
→ LLM 生成
```

但当前演示版不需要。

## 10. 调优方法

如果现场发现推荐不符合预期，优先从以下位置调：

1. `data/policies.json`
   - 补充政策关键词 `keywords`。
   - 优化 `target` 和 `script_hint`。
   - 检查政策分类是否能被规则识别。

2. `src/matcher.py`
   - 调整某类场景的加分权重。
   - 增加新的投诉关键词。
   - 增加新的政策类型识别函数。

3. `src/llm_client.py`
   - 优化 Prompt，强调推荐顺序、话术风格、风险提示。
   - 但不要把完整手机号或敏感信息传给 LLM。

4. `src/templates.py`
   - 优化本地兜底话术。
   - 保证无 API Key 时仍能演示完整链路。

## 11. 一个例子

客户输入：

```text
我这个套餐越来越贵，别人新用户比我便宜多了，不行我就携号转网了。
```

画像：

```text
月租 129 元，存量客户，网龄 6 年，有宽带，家庭号码 3 个，明确携转
```

本地规则会倾向于：

- 识别“携号转网”为高风险保有场景。
- 识别 129 元为中高套客户。
- 识别网龄 6 年为老客户保有。
- 识别有宽带和家庭号码，适合家庭融合或宽带权益。

因此保有类、返费类、宽带融合类政策会得分更高。随后 LLM 只在这些候选政策中选择 Top 3，并生成客服可直接使用的话术。
