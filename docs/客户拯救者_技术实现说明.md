# The Client Savior｜客户拯救者 技术实现说明

## 1. 项目定位

**客户拯救者** 是一个面向运营商一线投诉处理与客户挽留场景的轻量智能体 Demo。

它的目标不是建设完整客服系统，而是在一天开发周期内完成一个可演示、可运行、能体现 LLM 核心价值的 MVP：

```text
输入客户投诉和基础画像
        ↓
自动识别客户真实诉求与离网风险
        ↓
从本地政策库中召回候选政策
        ↓
调用云端大模型生成个性化挽留方案
        ↓
输出推荐政策、客服话术、办理提醒
```

本项目适合在普通笔记本电脑上运行。由于设备无独立显卡，系统不使用本地大模型，不使用向量数据库，不使用重型 Agent 框架，而是采用“本地轻量规则 + 云端 LLM + 模板兜底”的架构。

---

## 2. MVP 功能范围

### 2.1 已覆盖功能

1. **客户投诉录入**
   - 输入客户原话。
   - 填写当前月租、客户类型、网龄、家庭号码数等基础画像。
   - 勾选是否已有宽带、是否有换机需求、是否明确想离网。

2. **演示案例填充**
   - 支持从 `data/demo_cases.json` 读取演示案例。
   - 一键填充投诉内容和客户画像。

3. **本地政策召回**
   - 从 `data/policies.json` 读取政策卡。
   - 根据投诉关键词和客户画像召回 Top 5 候选政策。

4. **云端 LLM 生成**
   - 使用 OpenAI-compatible 接口调用云端模型。
   - 默认模型为 `qwen-turbo`。
   - 输出客户分析、推荐政策、挽留话术和内部提醒。

5. **本地模板兜底**
   - 未配置 API Key、网络异常、模型超时或 JSON 解析失败时，自动切换本地模板兜底。
   - 确保现场演示不中断。

6. **管理看板**
   - 统计已处理投诉数、高风险客户数、云端模型生成次数、本地模板兜底次数、平均耗时。
   - 展示投诉类型分布和推荐政策次数。

7. **政策库浏览**
   - 搜索政策名、关键词、适用场景。
   - 按政策分类筛选。
   - 展示核心权益、办理条件、风险提醒和话术提示。

## 3. 技术选型

| 模块 | 技术 | 选择原因 |
|---|---|---|
| 页面 | Streamlit | 开发快，适合一天内完成可演示 Demo |
| LLM 调用 | OpenAI SDK compatible mode | 方便接入 DashScope / 阿里云百炼 / 兼容接口 |
| 模型 | Qwen Turbo 或同类云端模型 | 响应快，适合 CPU 笔记本演示 |
| 政策库 | JSON 静态文件 | 稳定、启动快、避免现场解析 Word/Excel 出错 |
| 本地召回 | 规则打分 | 快速、可解释、无需向量库 |
| 兜底 | Python 模板 | 保证 API 异常时仍可演示 |
| 测试 | smoke_test.py | 快速验证数据、召回、兜底和 mock LLM 链路 |

## 6. 端到端处理流程

### 6.1 用户操作流程

1. 用户打开 Streamlit 页面。
2. 选择一个演示案例，或手动输入客户投诉。
3. 填写客户基础画像。
4. 点击“生成挽留方案”。
5. 页面展示：
   - 当前模式：云端模型生成 / 本地模板兜底。
   - 投诉类型、情绪、风险等级。
   - 推荐政策 Top 3。
   - 四段式挽留话术。
   - 内部办理提醒。

### 6.2 系统内部流程

```text
1. app.py 读取输入
2. 构造 CustomerProfile
3. 调用 ClientSaviorAgent.run()
4. matcher.match_policies() 从政策库召回 Top 5
5. 如果启用云端模型且 Key 有效：
   5.1 llm_client.compact_policy() 压缩候选政策
   5.2 构造 System Prompt + User Prompt
   5.3 调用 OpenAI-compatible Chat Completion
   5.4 解析 JSON 输出
   5.5 与本地政策详情合并
6. 如果云端模型失败：
   6.1 templates.build_fallback_result() 生成兜底方案
   6.2 插入失败原因提示
7. 返回 AgentResult
8. app.py 渲染结果并更新管理看板
```

## 8. 本地政策召回设计

### 8.1 召回目标

本地召回不是最终决策，而是给 LLM 提供安全、可控、可解释的候选政策范围。

这样做有三个好处：

1. **速度快**：纯 Python 规则匹配，毫秒级完成。
2. **可控**：LLM 只能从候选政策 ID 中选择，避免编造政策。
3. **可解释**：每个候选政策都有本地匹配原因和分数。

### 8.2 打分维度

`matcher.py` 主要根据以下维度打分：

| 维度 | 示例 |
|---|---|
| 关键词命中 | 投诉中出现“流量、宽带、携转、换机”等 |
| 离网风险 | 客户明确说“携号转网、换别家、不用了” |
| 当前月租 | 低资费客户适合流量卡，中高档客户适合保有或融合政策 |
| 宽带诉求 | 投诉涉及 WiFi、网速、上网课 |
| 终端诉求 | 客户有换机、购机、分期需求 |
| 商客场景 | 商户、门店、工作号、宣传触达 |
| 家庭号码数 | 家庭号码较多适合亲情网或家庭融合 |
| 网龄 | 老客户适合保有权益 |

### 8.3 输出结果

本地召回返回 Top 5：

```python
PolicyMatch(
    policy_id="P01",
    title="39元新潮玩乐享卡",
    score=80,
    reasons=["命中关键词：流量、不够", "当前套餐较低，适合大流量方案"],
    policy={...}
)
```

## 9. LLM 设计

### 9.1 调用方式

系统使用 OpenAI-compatible Chat Completion 接口。默认配置：

```text
LLM_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
LLM_MODEL=qwen-turbo
LLM_TIMEOUT=6
LLM_MAX_TOKENS=800
```

环境变量支持：

```text
DASHSCOPE_API_KEY
OPENAI_API_KEY
LLM_BASE_URL
LLM_MODEL
LLM_TIMEOUT
LLM_MAX_TOKENS
```

### 9.2 Prompt 约束

System Prompt 对模型提出以下约束：

1. 只允许从候选政策 ID 中选择。
2. 不允许编造政策或产品。
3. 推荐政策最多 3 个。
4. 涉及办理资格、合约、优惠期、预存、初装费、调测费时，必须提示“以系统办理结果为准”。
5. 话术要像一线客服：先安抚，再解释，再引导办理。
6. 输出必须是合法 JSON。

### 9.3 输入压缩

为保证响应速度，不把完整政策原文发送给模型。

`compact_policy()` 只保留：

```text
id, title, category, price, target, benefits, conditions, risk_notes, script_hint
```

列表字段最多保留前 4 项，长文本字段截断，避免 Prompt 过长。

### 9.4 JSON 输出结构

模型需要输出：

```json
{
  "customer_analysis": {
    "complaint_type": "资费争议",
    "emotion": "不满",
    "risk_level": "高",
    "key_needs": ["降低月租", "老客户优惠", "避免携转"],
    "summary": "客户认为老用户资费不公平，有携转风险。"
  },
  "recommended_policies": [
    {
      "policy_id": "P05",
      "rank": 1,
      "reason": "客户为中高套老客户，对资费不满且有携转风险。",
      "talking_point": "可优先核查老客户返费保有权益。"
    }
  ],
  "retention_script": {
    "opening": "您好，您反馈的老客户资费问题我理解，我先帮您看是否有保有权益。",
    "solution": "结合您的套餐和网龄，可以优先核查返费或融合类政策。",
    "risk_disclosure": "办理资格、合约、优惠期、预存、初装费、调测费以系统办理结果为准。",
    "next_step": "我先帮您查一下当前号码是否符合办理条件。"
  },
  "internal_notes": ["先核查互斥合约", "不要承诺最终资费，需以系统为准"]
}
```

### 9.5 响应解析与缓存

`llm_client.py` 做了以下稳定性处理：

1. 优先使用 `response_format={"type":"json_object"}`。
2. 如果兼容接口不支持 JSON 模式，自动重试普通调用。
3. 支持解析纯 JSON、Markdown code block 包裹的 JSON，以及前后带少量说明文字的 JSON。
4. 使用投诉内容、客户画像、候选政策 ID 生成缓存 Key。
5. 命中缓存时直接返回，耗时置为 0。

## 10. 本地模板兜底设计

### 10.1 触发场景

以下情况会进入本地模板兜底：

- 未配置有效 API Key。
- 云端 LLM 网络异常。
- 云端 LLM 超时。
- 云端 LLM 返回内容无法解析为 JSON。
- LLM 推荐了不存在的政策 ID。

### 10.2 兜底内容

`templates.py` 会根据轻量规则生成：

- 投诉类型
- 情绪
- 风险等级
- 关键诉求
- 推荐政策 Top 3
- 四段式挽留话术
- 内部提醒

这保证现场即使没有网络，也能完成完整演示。

## 12. 性能设计

### 12.1 性能目标

| 场景 | 目标耗时 |
|---|---|
| 本地政策召回 | < 1 秒 |
| 本地模板兜底 | < 1 秒 |
| 云端 LLM 正常调用 | 3~8 秒 |
| 云端 LLM 异常 | 超时时间内自动兜底，默认 6 秒 |

### 12.2 为什么不本地跑大模型

当前演示设备是无独立显卡笔记本。即使 Ultra 9 CPU 性能较强，本地跑 7B 级模型仍可能出现：

- 首次加载慢。
- 单次推理慢。
- 内存占用大。
- 现场不可控。

因此 MVP 选择云端 LLM，CPU 只负责页面、本地匹配和兜底。

### 12.3 控制耗时的方法

1. 候选政策只传 Top 5。
2. 每张政策只传精简字段。
3. 限制 `max_tokens`。
4. 低温度输出，减少发散。
5. 6 秒超时兜底。
6. 相同输入命中缓存。

## 15. 演示话术建议

### 15.1 开场

> 这个产品叫“客户拯救者”，面向一线客服和客户经理。它解决的问题是：客户投诉来的时候，一线人员需要快速判断客户真实诉求、离网风险，并从复杂政策中找到合适的挽留方案。

### 15.2 展示核心流程

> 我们不直接让大模型自由发挥，而是先用本地政策库召回候选政策，再让云端模型在候选范围内生成方案。这样既有 AI 的理解和话术生成能力，也能避免模型编造不存在的政策。

### 15.3 展示兜底能力

> 如果现场网络或者 API 出现问题，系统会在超时后自动切换本地模板兜底，确保一线人员仍然能拿到可执行方案。

### 15.4 收尾

> 后续如果接入真实 CRM、BOSS 系统和工单系统，就可以把“推荐政策”和“办理资格核查”进一步自动化。

## 16. 风险与后续扩展

### 16.1 当前风险

| 风险 | 影响 | 缓解方案 |
|---|---|---|
| API Key 未配置 | 无法云端生成 | 页面显示状态，支持本地兜底 |
| 网络不稳定 | LLM 调用失败 | 6 秒超时后兜底 |
| LLM 返回格式异常 | 结果无法解析 | JSON 解析增强，失败后兜底 |
| 政策数据不完整 | 推荐准确性有限 | 增加政策卡，接入真实政策库 |
| 办理资格不确定 | 可能误导客户 | 全部提示以系统办理结果为准 |
